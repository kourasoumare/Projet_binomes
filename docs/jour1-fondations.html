Jour 1 - Fondations Node.js et CLI gamifiée

Objectif du jour : comprendre Node.js, maîtriser les bases JavaScript côté serveur, puis livrer une version CLI d'un mini-jeu.

Fil rouge gamifié : Campus Quest, un jeu de quêtes avec XP et niveaux.

Les mots soulignés sont définis au survol (ou au focus clavier).

1. Introduction historique de Node.js

Node.js apparaît en 2009 avec une idée simple : utiliser JavaScript sur le serveur pour gérer efficacement les opérations d'entrée/sortie (réseau, fichiers, API).

  • 2009 : premières versions, centrées sur la performance I/O.
  • 2010 : npm accélère l'écosystème et le partage de packages.
  • Aujourd'hui : runtime mature, versions LTS stables, usage massif en production.

4. Comprendre Node.js simplement

Concepts à retenir

  • Node.js exécute JavaScript hors navigateur.
  • Node.js est adapté aux applications I/O (fichiers, réseau, API).
  • L'event loop gère les opérations asynchrones sans bloquer l'exécution.

Point fondamental : comprendre l'event loop

L'event loop est le mécanisme central de Node.js : elle orchestre ce qui s'exécute maintenant (call stack) et ce qui s'exécutera plus tard (callbacks, timers, I/O, microtasks). Sans ce modèle, il est impossible de comprendre pourquoi un serveur Node reste réactif sous charge I/O.

Comment lire ce schéma

  1. Call Stack : le code synchrone s'exécute ici, une fonction à la fois.
  2. libuv + OS : les tâches I/O (fichier, réseau, DNS) sont déléguées hors du thread principal.
  3. Event Loop : elle récupère les callbacks prêts et les injecte dans la stack.
  4. Microtasks : après chaque callback, Node traite d'abord Promises et process.nextTick.

Ordre d'exécution à connaître

console.log("A");
setTimeout(() => console.log("B timeout"), 0);
Promise.resolve().then(() => console.log("C promise"));
console.log("D");

// Résultat :
// A
// D
// C promise
// B timeout

Pourquoi ? Le synchrone passe d'abord (A, D), puis les microtasks (Promise), puis les macrotasks (timeout).

Liste claire : microtasks vs macrotasks

Microtasks (prioritaires) :

  • Promise.then/catch/finally
  • await (la suite après résolution de la Promise)
  • queueMicrotask(...)

Macrotasks (tour suivant de boucle) :

  • setTimeout(...) même avec 0
  • setInterval(...)
  • callbacks I/O (fichiers, réseau)
  • setImmediate(...) (Node.js)

Note Node.js : process.nextTick(...) est encore prioritaire (file spéciale) et passe avant les microtasks Promise.

Illustration : un tour de boucle

Lecture rapide : a chaque tour, Node prend une tache prete, execute son callback, vide les microtasks, puis passe au tour suivant. La duree n est pas fixe.

Piège classique : bloquer la boucle

const start = Date.now();

Promise.resolve().then(() => console.log("Promise exécutée"));

while (Date.now() - start < 3000) {
  // boucle CPU bloquante
}

console.log("Fin sync");

// Sortie (après ~3s) :
// Fin sync
// Promise exécutée

Même une Promise est retardée tant que la boucle CPU tourne : le thread principal est bloqué, donc l'event loop ne peut pas traiter les microtasks.

Bonnes pratiques immédiates

  • Éviter les calculs CPU lourds sur le thread principal.
  • Privilégier les APIs asynchrones (fs/promises, appels réseau async).
  • Découper les traitements longs en tâches plus petites.
  • Pour le CPU intensif, envisager worker_threads ou un service séparé.

5. Exemples

Exemple 1 - Lecture JSON simple

import { readFile } from "node:fs/promises";

const raw = await readFile("./data/players.json", "utf8");
const players = JSON.parse(raw);
console.log("Nombre de joueurs:", players.length);

Exemple 2 - Lecture de fichier + calcul de moyenne

Fichier scores.csv attendu :

pseudo;score
alice;12
bob;18
chris;10
import { readFile } from "node:fs/promises";

const csv = await readFile("./data/scores.csv", "utf8");
const lines = csv.trim().split("\n").slice(1);
const scores = lines.map(line => Number(line.split(";")[1]));
const moyenne = scores.reduce((acc, n) => acc + n, 0) / scores.length;
console.log("Moyenne:", moyenne.toFixed(2));

Exemple 3 - Statistiques simples (min, max, moyenne)

function stats(values) {
  const total = values.reduce((acc, n) => acc + n, 0);
  return {
    min: Math.min(...values),
    max: Math.max(...values),
    avg: total / values.length
  };
}

console.log(stats([12, 18, 10, 16]));

Exemple 4 - Gestion d'erreur fichier absent

import { readFile } from "node:fs/promises";

async function readJsonSafe(path) {
  try {
    const raw = await readFile(path, "utf8");
    return JSON.parse(raw);
  } catch (err) {
    if (err.code === "ENOENT") return [];
    throw err;
  }
}

6. Structurer Campus Quest (CLI)

C'est quoi Campus Quest ?

Campus Quest est un mini-jeu en ligne de commande. Vous devez coder une CLI qui manipule deux fichiers JSON (players.json et quests.json) pour simuler la progression d'un joueur.

Principe : un joueur accepte des quêtes, les valide, gagne de l'XP, puis monte de niveau.

Qu'est-ce qu'une quête ? Une quête est une mission avec un identifiant, un intitulé et une récompense d'XP. Exemple: q-intro-io = "Lire un fichier JSON" = 20 XP.

Différence importante :

  • quests.json = catalogue global des quêtes disponibles (source de vérité).
  • players.json = progression de chaque joueur (quêtes acceptées/terminées, XP, niveau).

Ce que vous devez faire concrètement

  1. Créer un joueur (pseudo unique).
  2. Lui attribuer une ou plusieurs quêtes existantes.
  3. Valider une quête pour ajouter de l'XP.
  4. Afficher ses stats (XP, niveau, quêtes) puis un classement simple.

Entrée : commandes terminal (ex: node src/cli.js create-player Alice).
Sortie : messages CLI lisibles + données persistées correctement dans les JSON.

Exemple de données attendues

// quests.json
[
  { "id": "q-intro-io", "title": "Lire un fichier JSON", "difficulty": "easy", "xp": 20 },
  { "id": "q-api-1", "title": "Créer une route GET /players", "difficulty": "medium", "xp": 40 }
]

// players.json
[
  {
    "id": "p-alice",
    "pseudo": "Alice",
    "xp": 20,
    "level": 1,
    "acceptedQuests": ["q-intro-io"],
    "completedQuests": ["q-intro-io"]
  }
]

Lecture du scénario: Alice accepte la quête q-intro-io, la termine, passe à 20 XP, et sa progression est enregistrée dans players.json.

Arborescence cible

campus-quest/
  package.json
  src/
    cli.js
    store.js
    game-service.js
  data/
    players.json
    quests.json

Rôle précis de chaque fichier

  • package.json : configuration du projet (type ESM, scripts npm, dépendances).
  • src/cli.js : point d'entrée terminal; lit process.argv, route la commande, affiche le résultat.
  • src/game-service.js : logique métier (création joueur, attribution/validation de quête, calcul XP/niveau).
  • src/store.js : accès aux fichiers JSON (lecture/écriture, gestion d'erreurs I/O).
  • data/quests.json : catalogue des quêtes disponibles (id, titre, difficulté, XP).
  • data/players.json : état des joueurs (pseudo, XP, niveau, quêtes acceptées/terminées).

Règle d'architecture : cli.js ne contient pas les règles de jeu; il appelle game-service.js, qui lui-même passe par store.js pour persister les données.

Version concrète (exemple) pour node src/cli.js create-player Alice :

  1. cli.js lit la commande create-player et le pseudo Alice.
  2. cli.js appelle createPlayer("Alice") dans game-service.js.
  3. game-service.js vérifie les règles (pseudo valide, unicité, etc.).
  4. store.js lit puis écrit players.json.
  5. cli.js affiche "Joueur créé".

Fonctions à implémenter

  • createPlayer(pseudo)
  • acceptQuest(playerId, questId)
  • completeQuest(playerId, questId)
  • showStats(playerId)

Exemple simple guidé : premier cli.js

Objectif : comprendre le rôle de cli.js avant la version complète Campus Quest.

const [, , command, name] = process.argv;

if (command === "hello") {
  console.log(`Bonjour ${name ?? "inconnu"}`);
} else if (command === "help" || !command) {
  console.log("Usage: node src/cli.js hello <prenom>");
} else {
  console.log(`Commande inconnue: ${command}`);
  process.exitCode = 1;
}

Pourquoi [, , command, name] ? Les deux premières valeurs de process.argv ne sont pas des arguments métier :

  • process.argv[0] = chemin vers l'exécutable Node.
  • process.argv[1] = chemin vers le fichier exécuté (src/cli.js).
  • process.argv[2] = votre première vraie donnée utilisateur (ici command).
  • process.argv[3] = la suivante (ici name).

Test rapide :

node src/cli.js help
node src/cli.js hello Alice
node src/cli.js test

Ce pattern (lire process.argv → router la commande → afficher le résultat) est exactement celui de Campus Quest.

Lire et écrire un fichier JSON (indispensable)

Dans Campus Quest, tout passe par store.js. Voici la base minimale attendue :

import { readFile, writeFile } from "node:fs/promises";

export async function loadJson(path) {
  try {
    const raw = await readFile(path, "utf8");
    return JSON.parse(raw);
  } catch (err) {
    if (err.code === "ENOENT") return [];
    throw err;
  }
}

export async function saveJson(path, data) {
  const pretty = JSON.stringify(data, null, 2);
  await writeFile(path, `${pretty}\n`, "utf8");
}

Exemple d'utilisation dans une commande :

const players = await loadJson("./data/players.json");
players.push({ id: "p-alice", pseudo: "Alice", xp: 0, level: 1 });
await saveJson("./data/players.json", players);

À retenir : on lit d'abord, on modifie en mémoire, puis on réécrit le fichier.

TermeDéfinition

I/OEntrées/sorties : fichiers, réseau, terminal, disque.

MacrotaskTâche exécutée ensuite (ex: setTimeout).
