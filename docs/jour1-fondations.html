Jour 1 - Fondations Node.js et API JSON

Objectif du jour : comprendre Node.js, maîtriser les bases JavaScript côté serveur, puis livrer une API JSON minimale dans sandbox-server.

Fil rouge technique : sandbox-server, une API Node.js minimale en JSON.

Les mots soulignés sont définis au survol (ou au focus clavier).

1. Introduction historique de Node.js

Node.js apparaît en 2009 avec une idée simple : utiliser JavaScript sur le serveur pour gérer efficacement les opérations d'entrée/sortie (réseau, fichiers, API).

  • 2009 : premières versions, centrées sur la performance I/O.
  • 2010 : npm accélère l'écosystème et le partage de packages.
  • Aujourd'hui : runtime mature, versions LTS stables, usage massif en production.

4. Comprendre Node.js simplement

Concepts à retenir

  • Node.js exécute JavaScript hors navigateur.
  • Node.js est adapté aux applications I/O (fichiers, réseau, API).
  • L'event loop gère les opérations asynchrones sans bloquer l'exécution.

Point fondamental : comprendre l'event loop

L'event loop est le mécanisme central de Node.js : elle orchestre ce qui s'exécute maintenant (call stack) et ce qui s'exécutera plus tard (callbacks, timers, I/O, microtasks). Sans ce modèle, il est impossible de comprendre pourquoi un serveur Node reste réactif sous charge I/O.

Comment lire ce schéma

  1. Call Stack : le code synchrone s'exécute ici, une fonction à la fois.
  2. libuv + OS : les tâches I/O (fichier, réseau, DNS) sont déléguées hors du thread principal.
  3. Event Loop : elle récupère les callbacks prêts et les injecte dans la stack.
  4. Microtasks : après chaque callback, Node traite d'abord Promises et process.nextTick.

Ordre d'exécution à connaître

console.log("A");
setTimeout(() => console.log("B timeout"), 0);
Promise.resolve().then(() => console.log("C promise"));
console.log("D");

// Résultat :
// A
// D
// C promise
// B timeout

Pourquoi ? Le synchrone passe d'abord (A, D), puis les microtasks (Promise), puis les macrotasks (timeout).

Différence simple : microtasks vs macrotasks

Microtask = priorité haute, exécutée juste après le code courant.
Macrotask = exécutée ensuite (pas immédiatement).

TypeAPIs typiquesQuand ça passe
MicrotaskPromise.then/catch/finally, await, queueMicrotaskAvant les macrotasks
MacrotasksetTimeout, setInterval, callbacks I/O, setImmediateAprès les microtasks

Note Node.js : process.nextTick(...) est encore plus prioritaire que les microtasks Promise.

Piège classique : bloquer la boucle

const start = Date.now();

Promise.resolve().then(() => console.log("Promise exécutée"));

while (Date.now() - start < 3000) {
  // boucle CPU bloquante
}

console.log("Fin sync");

// Sortie (après ~3s) :
// Fin sync
// Promise exécutée

Même une Promise est retardée tant que la boucle CPU tourne : le thread principal est bloqué, donc l'event loop ne peut pas traiter les microtasks.

Bonnes pratiques immédiates

  • Éviter les calculs CPU lourds sur le thread principal.
  • Privilégier les APIs asynchrones (fs/promises, appels réseau async).
  • Découper les traitements longs en tâches plus petites.
  • Pour le CPU intensif, envisager worker_threads ou un service séparé.

5. Exercice unique - Campus Quest en REST API (sans CLI)

Objectif : transformer le jeu Campus Quest en API REST tout en gardant la persistance des données en fichiers JSON dans data/.

Contrainte : aucune logique CLI, tout passe par des routes HTTP JSON.

Structure minimale attendue

sandbox-server/
  src/
    server.js
  data/
    players.json
    enigmes.json
    attempts.json

Routes à implémenter

  1. POST /players
  2. GET /enigmes (sans answer)
  3. POST /enigmes/:id/answer
  4. GET /leaderboard
  5. GET /health

Règles métier

  • Une énigme validée donne de l'XP au joueur.
  • Une même énigme ne doit pas redonner de l'XP au même joueur.
  • Le classement est trié par XP décroissant.
  • Les données doivent rester persistées après redémarrage du serveur.

Critères de réussite

  • Les 5 routes répondent en JSON avec des statuts HTTP cohérents.
  • Aucune route ne dépend d'une commande terminal applicative.
  • Le dossier data/ contient l'état du jeu (joueurs, énigmes, tentatives).
