1. Introduction historique de Node.js

Node.js apparaît en 2009 avec une idée simple : utiliser JavaScript sur le serveur pour gérer efficacement les opérations d'entrée/sortie (réseau, fichiers, API).

  • 2009 : premières versions, centrées sur la performance I/O.
  • 2010 : npm accélère l'écosystème et le partage de packages.
  • Aujourd'hui : runtime mature, versions LTS stables, usage massif en production.

4. Comprendre Node.js simplement

Concepts à retenir

  • Node.js exécute JavaScript hors navigateur.
  • Node.js est adapté aux applications I/O (fichiers, réseau, API).
  • L'event loop gère les opérations asynchrones sans bloquer l'exécution.

Point fondamental : comprendre l'event loop

L'event loop est le mécanisme central de Node.js : elle orchestre ce qui s'exécute maintenant (call stack) et ce qui s'exécutera plus tard (callbacks, timers, I/O, microtasks). Sans ce modèle, il est impossible de comprendre pourquoi un serveur Node reste réactif sous charge I/O.

Comment lire ce schéma

  1. Call Stack : le code synchrone s'exécute ici, une fonction à la fois.
  2. libuv + OS : les tâches I/O (fichier, réseau, DNS) sont déléguées hors du thread principal.
  3. Event Loop : elle récupère les callbacks prêts et les injecte dans la stack.
  4. Microtasks : après chaque callback, Node traite d'abord Promises et process.nextTick.

Ordre d'exécution à connaître

console.log("A");
setTimeout(() => console.log("B timeout"), 0);
Promise.resolve().then(() => console.log("C promise"));
console.log("D");

// Résultat :
// A
// D
// C promise
// B timeout

Pourquoi ? Le synchrone passe d'abord (A, D), puis les microtasks (Promise), puis les macrotasks (timeout).

Piège classique : bloquer la boucle

const start = Date.now();
while (Date.now() - start < 3000) {
  // boucle CPU bloquante
}
console.log("Fin");

Pendant cette boucle, Node ne peut pas traiter les autres requêtes : le serveur devient lent ou indisponible.

Bonnes pratiques immédiates

  • Éviter les calculs CPU lourds sur le thread principal.
  • Privilégier les APIs asynchrones (fs/promises, appels réseau async).
  • Découper les traitements longs en tâches plus petites.
  • Pour le CPU intensif, envisager worker_threads ou un service séparé.

5. Exemples

Exemple 1 - Lecture JSON simple

import { readFile } from "node:fs/promises";

const raw = await readFile("./data/players.json", "utf8");
const players = JSON.parse(raw);
console.log("Nombre de joueurs:", players.length);

Exemple 2 - Lecture de fichier + calcul de moyenne

Fichier scores.csv attendu :

pseudo;score
alice;12
bob;18
chris;10
import { readFile } from "node:fs/promises";

const csv = await readFile("./data/scores.csv", "utf8");
const lines = csv.trim().split("\n").slice(1);
const scores = lines.map(line => Number(line.split(";")[1]));
const moyenne = scores.reduce((acc, n) => acc + n, 0) / scores.length;
console.log("Moyenne:", moyenne.toFixed(2));

Exemple 3 - Statistiques simples (min, max, moyenne)

function stats(values) {
  const total = values.reduce((acc, n) => acc + n, 0);
  return {
    min: Math.min(...values),
    max: Math.max(...values),
    avg: total / values.length
  };
}

console.log(stats([12, 18, 10, 16]));

Exemple 4 - Gestion d'erreur fichier absent

import { readFile } from "node:fs/promises";

async function readJsonSafe(path) {
  try {
    const raw = await readFile(path, "utf8");
    return JSON.parse(raw);
  } catch (err) {
    if (err.code === "ENOENT") return [];
    throw err;
  }
}

11. Glossaire

TermeDéfinition
RuntimeProgramme qui exécute un langage (Node.js exécute JavaScript).
I/OEntrées/sorties : fichiers, réseau, terminal, disque.
Event loopBoucle qui orchestre les tâches asynchrones en Node.js.
Call stackPile d'exécution synchrone des fonctions.
MicrotaskTâche prioritaire exécutée après le synchrone (ex: Promise).
MacrotaskTâche exécutée ensuite (ex: setTimeout).
Thread poolPool de threads utilisé par libuv pour certaines opérations.

JSONFormat texte structuré utilisé pour stocker/échanger des données.
